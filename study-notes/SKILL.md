# 作業複習筆記（study-notes）

把一份「已完成的作業/練習專案」整理成可長期回顧的結構化筆記。核心價值：寫完當下腦中最清楚，趁記憶新鮮把「這在學什麼、關鍵語法、踩過的坑」固化下來，日後看筆記就能一目瞭然，不用重讀整個專案。

## 產出總覽

先問使用者要哪種格式，再產出。**內容相同，只是容器不同**:

| 格式 | 產出 | 適合 |
|---|---|---|
| **四檔資料夾** | `複習筆記/` 內 README + 01 + 02 + 03 | 在 VS Code / GitHub 上看，分檔清楚 |
| **HackMD 單檔** | `複習筆記/hackmd-單檔版.md`（可另附書本目錄） | 貼進 HackMD、想線上閱讀/分享 |
| **兩者都建** | 以上兩者都產出 | 本機分檔 + 線上單檔，一次到位 |

四檔的角色（HackMD 單檔版把這三塊合併成一篇）：

| 檔案 | 角色 | 什麼時候看 |
|---|---|---|
| `README.md` | 索引 | 進入點，先看它決定翻哪一份 |
| `01-總覽與流程圖.md` | 全貌 | 想快速回憶「這在學什麼、怎麼運作」 |
| `02-知識點速查.md` | 速查 | 寫程式忘記語法時查 |
| `03-我的錯誤筆記.md` | 避雷 | 下次寫類似作業前先掃一遍 |

骨架模板放在 `references/note-templates.md`（四檔模板 + HackMD 單檔版模板 + 書本目錄模板），直接套用可確保結構一致。

## 執行流程

### 步驟 1：確認作業資料夾

使用者若指定路徑就用之；否則用當前工作目錄。先列出檔案（`ls`）掌握全貌，確認這是一個作業/專案資料夾。若已存在 `複習筆記/`，詢問使用者要覆蓋更新還是新增（避免蓋掉他手動補充的內容）。

### 步驟 2：詢問筆記格式（先問再動手）

用 AskUserQuestion 讓使用者選，**三個選項**:

- **四檔資料夾** — `複習筆記/` 內 README + 01 + 02 + 03，適合在 VS Code / GitHub 上看。
- **HackMD 單檔** — 一份合併 `.md`，適合貼進 HackMD 線上閱讀/分享。
- **兩者都建**（推薦） — 本機四檔 + HackMD 單檔都產出。

這一步決定後面「步驟 4 怎麼產出」，但**步驟 3 的探索理解與內容素材兩種格式共用**，不用做兩次。

### 步驟 3：探索與理解（這是筆記品質的關鍵，格式無關）

目標是搞懂四件事，才有辦法寫出對的筆記：

1. **技術主題** — 這是什麼語言/框架/領域？先看依賴與設定檔判斷（如 `package.json`、`requirements.txt`、`go.mod`、`Cargo.toml`、`pom.xml`），再看主要原始碼。**不要預設是任何特定技術**，要依實際專案而定。
2. **教學目的** — 這份作業想訓練什麼能力？看說明檔（README、作業說明、註解裡的 TODO）最快。
3. **任務拆解** — 作業被切成哪幾個部分/功能？通常對應檔案結構或 TODO 編號。
4. **使用者實際寫了什麼** — 讀他完成的原始碼，理解他的解法。

讀取順序建議：說明檔 → 依賴/設定檔 → 主要原始碼 → 測試檔（測試檔常常是「規格」的最佳來源）。

### 步驟 4：依格式產出

把步驟 3 得到的素材（總覽/知識點/錯誤三大塊）先想清楚，再依所選格式渲染。

**A. 四檔資料夾** — 依 `references/note-templates.md` 的四檔骨架，在 `複習筆記/` 建立：

- **README（索引）** — 一句話總結整份作業、三份筆記的導覽表、操作流程（怎麼跑/怎麼驗收）。四個檔互相用相對連結串起來。
- **01 總覽與流程圖** — 「這份作業在做什麼（一句話）」、教學目的表、專案結構分析、**ASCII 運作流程圖**、任務/功能對照表。流程圖至少畫兩張：一張「啟動/建置流程」，一張「一次典型操作的資料流」。
- **02 知識點速查** — 語法/API/觀念做成「忘記時查」的段落，搭配**最小可用的正確範例**。只放正確寫法；有陷阱用正向提醒點出。
- **03 我的錯誤筆記** — 錯誤總表（錯誤寫法 → 正確寫法）加逐項說明。這是最個人化、日後最有價值的一份。

**B. HackMD 單檔** — 依 `references/note-templates.md` 的 HackMD 模板產出。**流程**:

1. **先問「是否需要書本目錄？」**（AskUserQuestion 是/否）。
   - 「是」 → 在 `複習筆記/hackmd-書本目錄.md` 建立書本索引骨架（若已存在，先問是否覆蓋 —— 它是**跨週共用**的索引，可能已被手動維護過）。書本目錄通常**只建一次**，之後每週把新單篇的網址補一行即可。
2. **再建 `複習筆記/hackmd-單檔版.md`**，規格務必遵守：
   - **YAML frontmatter**：`title` 只寫「Week N · 作業主題」——**不加「複習筆記」**（書本標題已有，避免重複）；加 `tags`。
   - 開頭放 `**📑 目錄**` 一行，下面接 `[TOC]`。
   - 三大段（總覽與流程圖 / 知識點速查 / 我的錯誤筆記）用 **H1**，其下小節用 **H2**；**更深層的小標一律降成粗體文字**（讓 `[TOC]` 只列到 H2、乾淨可跳）。
   - 全形標點、ASCII **完整四邊框**流程圖（見 `formatting-rules.md`）。
   - **不要**放 `[← 回索引]`、`{#id}`、`<a id>` 這些（HackMD 會過濾錨點，而且單檔不需跨檔連結）。

**C. 兩者都建** — 先做 A，再做 B（內容一次想好、渲染兩種容器）。

### 步驟 5：錯誤筆記（03 / 錯誤段）的來源策略

素材依情境有兩種來源：

- **當前對話有除錯/踩坑紀錄** → 直接萃取：把使用者這次實際犯的錯、你指出的修正，整理成「錯誤 → 正確」對照。這是最理想的情況。
- **沒有對話紀錄（乾淨地重跑）** → 依程式碼與主題，推斷該主題的**常見易錯點**寫入，並明確標註為「常見陷阱」，邀請使用者補上自己實際遇到的錯誤。

### 步驟 6：內容分流原則（避免筆記互相干擾）

- **正確示範 → 知識點速查；錯誤對照 → 錯誤筆記。** 兩者分工，避免速查表混入錯誤寫法而誤導。
- **有語法/觀念學習點的錯誤**(對照著看能加深理解，例如「為什麼不能用 `!idx` 判斷」)→ 可在速查段用正向提醒帶到，完整對照放錯誤段。
- **純打錯字、無學習點的失誤**（例如屬性名拼錯）→ 一律只放錯誤段，不要污染正確示範。

### 步驟 7：套用格式慣例

嚴格遵守 `references/formatting-rules.md` 的規則：中文正文全形標點、程式碼半形、引用區塊斷行、**ASCII 流程圖用完整四邊框（區隔清楚；右框線遇中文略不齊可接受）**、以及 **HackMD 專用規則（用 `[TOC]`、禁用自訂錨點、標題不含「複習筆記」）**。這組規則對閱讀體驗影響很大，務必套用。

### 步驟 8：完成後（依格式給提示）

- **四檔** → 告訴使用者從 `複習筆記/README.md` 進入；日後又踩到新坑，往 `03-我的錯誤筆記.md` 總表加一列即可。
- **HackMD** → 提示使用者：用 VS Code 開 `hackmd-單檔版.md`,`Cmd+A` 全選複製，貼進一篇 HackMD note(貼之前若 VS Code 顯示的是舊版，先 `Cmd+Shift+P` → `Revert File` 從磁碟重載)。若有建書本目錄，把這篇的 HackMD 網址填進書本目錄那篇、開 Book mode 串起來。

## 心法

- **筆記是給「未來的自己」看的**，不是給改作業的老師。用最省力回憶的方式組織：能一眼掃到重點、忘記時查得到、避雷表掃一遍就好。
- **先問格式、再一次產出**：探索理解與內容素材是共用的，別因為要兩種格式就做兩次功。
- **通用優先**：先辨識技術主題，再決定流程圖與知識點的內容。同一套結構套用到 Web 後端、前端、演算法、資料處理、CLI 工具都成立——變的是內容，不是骨架。
- **閱讀體驗至上**:HackMD 用原生 `[TOC]`、乾淨的標題階層、不跑版的流程圖；四檔用相對連結互跳。細節都在 `formatting-rules.md`。
- **簡潔勝過詳盡**：速查段給最小可用範例即可，不要把整份原始碼搬進來。
